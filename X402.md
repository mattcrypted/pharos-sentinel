# Sentinel over x402 — pay-per-query risk assessment

Pharos's x402 guide names, as a flagship use case, *"supply chain risk assessment
queries: enterprises pay-per-query for supplier risks."* That is precisely what
Sentinel does — so Sentinel ships an optional **x402 paid-call gate** that turns
`risk_check` into a metered, pay-as-you-go Skill for the Pharos autonomous economy.

- Server: [`sentinel_x402.py`](sentinel_x402.py) — wraps `risk_check` behind x402.
- Client demo: [`x402_demo.py`](x402_demo.py) — runs the full loop on live Atlantic.
- Spec: this file.

## The flow

```
GET /risk_check?address=..&action=..            ->  402 Payment Required
                                                    PAYMENT-REQUIRED: <base64 PaymentRequired>

(client sends a micro-transfer to payTo on Atlantic, gets a tx hash)

GET /risk_check?address=..&action=..            ->  200 OK
   PAYMENT-SIGNATURE: <base64 settlement proof>     { verdict, score, reasons, data }
                                                     PAYMENT-RESPONSE: <base64 receipt+txHash>
```

This is the documented Pharos sequence: a `402` carrying a Base64 `PaymentRequired`
object in the `PAYMENT-REQUIRED` header, the client re-requesting with a
`PAYMENT-SIGNATURE` header, and the server returning `200` plus a `PAYMENT-RESPONSE`
header containing the settlement `tx_hash`.

A real run (live Atlantic, querying the `Backdoor` gallery fixture):

```
[1] unpaid           -> HTTP 402  pay 0.001 PHRS to 0xda5B…20f0 on eip155:688689
[2] pay              -> settled tx 0x4c36af4fd468d3b1467dc23fe0a31953387c0662f1f7bf9e4eb8c9def48f7d2d
[3] retry + proof    -> HTTP 200  verdict DANGEROUS (score 30)
[4] replay same tx   -> HTTP 402  payment already used (replay)
```

## Settlement model: native verify-by-RPC

x402 defines several settlement *schemes*. Sentinel implements the **native model
the Pharos docs describe directly**:

> *"the user sends a micro-token transfer to a specified address → the server
> verifies the transaction → the requested resource is automatically returned."*

The server verifies that settlement through **the same Foundry `cast` toolchain Sentinel
uses for risk** — `cast rpc eth_getTransactionByHash` + `eth_getTransactionReceipt` — checking:

1. the payment went **to** the advertised `payTo`,
2. its **value ≥ `maxAmountRequired`**,
3. the tx is **confirmed** (receipt status `0x1`),
4. the tx hash has **not been used before** (in-memory replay guard — the
   idempotency the Pharos guide recommends).

Two consequences that fit Sentinel's design exactly:

- **The server stays read-only and keyless.** It never signs or sends value — only
  the *client* does. Verification is pure read-only `cast` reads, so the server needs
  only the `payTo` **address**, never a private key.
- **No new dependencies.** The gate is stdlib `http.server` over the same
  `pharos_atlantic` Foundry `cast` helpers Sentinel already uses — it adds nothing
  beyond Foundry and the Python standard library. The chain-reader simply verifies
  its own payments.

`scheme` is advertised as `exact-native` and `network` as `eip155:688689` so a client
knows which settlement variant the server speaks.

## Relationship to the official `@x402` SDK

For full interop with off-the-shelf `@x402` **clients** (`wrapFetchWithPayment`), the
canonical Pharos path uses the EIP-3009 **`exact`** scheme — a signed
`transferWithAuthorization` payload verified/settled through a Facilitator — wired with
`@x402/express` + `viem`. See Pharos's reference skill:
`https://github.com/PharosNetwork/examples/tree/main/skills/x402-pharos`.

Sketch of the SDK route (production), gating the same Sentinel verdict:

```ts
import { paymentMiddleware, x402ResourceServer } from "@x402/express";
import { ExactEvmScheme } from "@x402/evm/exact/server";

app.use(paymentMiddleware({
  "GET /risk_check": {
    accepts: { scheme: "exact", price: "0.01", network: "eip155:688689", payTo },
    description: "Sentinel on-chain risk assessment (pay-per-query)",
  },
}, resourceServer));

app.get("/risk_check", (req, res) => res.json(riskCheck(req.query)));  // calls Sentinel
```

The trade-off was deliberate: the native-verify gate is **feasible** (no Node/
facilitator/test-USDC infrastructure), **testable** end-to-end with one funded testnet
key, and **protocol-compatible** at the HTTP envelope while preserving Sentinel's
read-only core. The `exact`-scheme SDK route is the drop-in upgrade for
production multi-client interop.

## Run it

```bash
python sentinel_x402.py     # serve the gate on 127.0.0.1:4021 (read-only)
python x402_demo.py         # 402 -> pay on Atlantic -> 200 verdict -> replay rejected
```

Price and recipient are configurable via `SENTINEL_PRICE_WEI` and `SENTINEL_PAY_TO`.
